# 项目经验提炼指南

`project-insights` skill 用于消费 `init` skill 已沉淀的项目架构文档，将分散的技术结论进一步提炼为更适合复盘、汇报和面试表达的项目经验材料。

## 触发方式

适合以下场景：

- 用户要求总结项目亮点、难点、经验沉淀。
- 用户希望把技术分析整理成更适合复盘或面试表达的内容。
- 用户需要输出项目价值、个人贡献或可复用经验。

示例提示词：

- “请根据已有项目文档提炼这个项目的亮点和难点。”
- “请把当前项目总结成适合面试表达的素材。”

## 阶段定位

这是整个技能集的第二阶段，依赖 `docs/project-architecture/`，并为 `interview-questions` 提供项目经验和表达素材。

## 输入前提

执行前优先检查以下文档是否存在：

- `docs/project-architecture/README.md`
- `docs/project-architecture/tech-stack.md`
- `docs/project-architecture/project-structure.md`
- `docs/project-architecture/engineering.md`

如果上述文档缺失较多，先提醒需要补齐 `init` 的分析结果，再继续在现有信息基础上谨慎输出，并明确标注“依据不足”。

## 输出要求

将内容统一沉淀到 `docs/project-insights/`。如果目录不存在则创建，已存在则增量更新。

默认产出以下文档：

- `docs/project-insights/README.md`：项目经验总览、提炼范围、核心结论。
- `docs/project-insights/highlights-and-challenges.md`：项目亮点、难点、解决策略、技术价值。
- `docs/project-insights/interview-storylines.md`：适合面试表达的项目经历、贡献总结和回答框架。

如果项目特别复杂，可以补充同目录下的新文档，但不要输出到 `docs/` 之外。

## 提炼重点

优先从已有文档中抽取以下内容：

1. 技术亮点：框架选型、组件体系、工程化能力、性能优化、复用设计、架构分层。
2. 实现难点：复杂状态、跨模块协作、权限控制、性能瓶颈、构建或部署复杂度。
3. 解决策略：如何拆解问题、采用了哪些方案、方案为何适配当前项目。
4. 项目经验：从交付效率、稳定性、可维护性、团队协作角度总结可复用经验。
5. 面试表述价值：哪些内容适合讲成“背景 - 挑战 - 行动 - 结果”。

## 执行流程

1. 先读取 `docs/project-architecture/` 下的已有文档，提取明确结论和可验证依据。
2. 按“技术亮点”“实现难点”“工程经验”“可讲述案例”分类整理信息。
3. 对每条提炼结果补充来源依据，避免脱离原始文档臆造结论。
4. 将内容整理成适合复盘与面试表达的文档，优先写清楚问题背景、具体动作和结果价值。
5. 对仍无法确认的亮点或难点，明确写成“待确认”，不要伪造业务结果。

## 文档写法

- 必须基于 `docs/project-architecture/` 中已沉淀的信息提炼。
- 可适度抽象表达，但不能脱离原始依据。
- 亮点要说明“为什么算亮点”，难点要说明“难在哪里”和“如何解决”。
- 面试表达要尽量口语化、结构化，适合直接转述。
- 不要编造量化指标；没有证据时使用“可能”“待确认”并写清判断依据。

## 推荐文档结构

### `README.md`

- 提炼范围
- 主要结论
- 推荐阅读顺序

### `highlights-and-challenges.md`

- 核心技术亮点
- 工程化亮点
- 项目难点
- 解决思路与价值

### `interview-storylines.md`

- 项目一句话介绍
- 可重点展开的 3 到 5 个经历
- 每个经历的回答框架
- 容易被追问的点

## 质量标准

- 文档应能直接支持项目复盘和面试准备。
- 提炼结论必须可追溯到已有架构文档。
- 相同观点避免重复表述，统一术语。
- 如果某项内容信息不足，要明确说明，而不是用通用套话填充。
